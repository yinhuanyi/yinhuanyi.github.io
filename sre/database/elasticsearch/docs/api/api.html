    # (二)Query DSL概念

    • Query查询规范和分类（这个很重要，了解查询规范后才知道什么时候使用什么类型的查询）
    基于JSON定义的查询语音，主要包括如下两种类型：
	1：字段类查询
		如：term（主要针对字段查询，不会分词处理）、match(主要针对text全文检索查询，能够分词处理)、query_string(也会被分词), range(范围查询)等，只针对某一个字段进行查询
		字段类的查询分为两类
			①：全文匹配：针对text类型的字段进行全文检索，会对查询语句进行分词处理，如match，match_phrase、query_string、simple_query_string等query类型
			②：单词匹配：不会对查询语句做分词处理，直接去匹配字段的倒排索引，如 term，terms，range等query类型，但是如果是查询text类型，那么term就会有问题，因为倒排索引中没有原始字段的值了
			
	2：复合查询
		如 bool 查询等，包含一个或多个字段查询或者复合查询语句

    # (三)相关性算分

    • 概念
    相关性算分是指文档与查询语句之间的相关程度，英文为：relevance
	通过倒排索引可以获取与查询语句相匹配的文档列表，那么如何将最符合用户的查询需求的文档放到最前列呢？
	本质上是一个排序问题，排序的依据是相关性算分


那么相关性算分取决于下面几个概念：
①：Term Frequency(TF)词频：即单词在该文档中出现的次数。词频越高，相关度越高
②：Document Frequency(DF)文档词频：即单词出现的文档数
③：Inverse Document Frequency(IDF)逆向文档频率，与文档频率相反，简单理解为1/DF, 即单词出现的文档数越少，相关度越高
④：Field-length Norm 文档越短，相关性越高


ES相关性算分模型主要有两类
①：TF/IDF 模型
②：BM25 模型 5.x之后默认的模型

    # (六)query string查询语句

    • 思考：已经有了match_phrase查询短语，为什么还需要有query_string呢？
    1：match_phrase查询：虽然查询的时候会被分词，但是顺序和位置的一致性，保证了match_phrase查询到的一定是短语。因此，查询 "nihao 你好" 不可以匹配到 "nihao 哈哈 你好"

2：query_string查询：不需要保证被分词后，每个词的顺序和位置，因此， 查询 "nihao 你好" 可以匹配到 "nihao 哈哈 你好"。query_string查询也可以随意添加操作符
    1
    2
    3

      "query": "python AND java"  # 这里AND表示，在job字段中两个都需要包含
    }
  }

    # 三：ES Search 操作的运行机制

    # (一) Query 和 Fetch

    • Search执行的时候实际上分为两个步骤运行
    1：Query 阶段
2：Fetch 阶段
    1
    2
    • 在Query 阶段查询的过程

    这里在查询的时候，在P0、P1、P2都会返回from+size个文档，一共返回了3倍的from+size个文档ID（注意，这里不是拿数据，而是获取ID列表），这些文档ID聚合的时候，根据排序值排序，最后在从3倍的from+size个文档ID中选择from+size个文档ID返回
    1
    • 在Fetch 阶段查询的过程

    Fetch阶段在获取到ID列表后，根据ID列表，查询每个Shard的详细数据，最后拼接数据，返回给用户，那么此时的search操作就结束了
    1

    # (二) 相关性算分的问题

    • ES的相关性算法在Shard和Shard之间是相互独立的，因此文档的_score与shard有关系，当查询的文档在不同shard的发布不均衡，那么_score的分数就会不准确

    • 可以通过explain关键字，获取每个查询的算法规则，可以知道搜索到的文档在哪个分配上

    GET /my_multi_index/_search
{
  "explain": true,
  "query": {
    "match": {
      "job": "python"
    }
  }
}
    1
