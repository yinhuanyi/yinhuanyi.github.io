    # 一：Kafka架构原理

    • Kafka体系结构逻辑图，一般而言再打的数据，一个topic有5个到10个的partition就足够了

    1：Broker：kafka的一个节点
2：Leader：在kafka集群中负责消息的接收和发送，当consumer消费不同的partition的时候，Leader可能是broker1、broker2、或者broker3，所以这里的Leader指的是partition的leader
3：Follower：复制Leader的数据，这里的Follower指的是partition的Follower

    # （一）基本概念

    • 概念
    1：Kafka它是一个分布式消息系统，由linkedin使用scala编写，用作LinkedIn的活动流（Activity Stream）和运营数据处理管道（Pipeline）的基础。具有高水平扩展和高吞吐量,Kafka支持仿AMQP协议，Kafka增加和减少服务器都会在Zookeeper节点上触发相应的事件kafka系统会捕获这些事件，进行新一轮的负载均衡，客户端也会捕获这些事件来进行新一轮的处理

2：Kafka相当于是一个broker集群，发送到每个topic的消息保存到一个或多个partition中，为了保证kafka消息的高可用，每个partition可以设置Replica的系数，但是为了节约系统资源，默认情况下Replica的系数为1
    1
    2
    3
    • Kafka中的逻辑角色功能
    1：Consumer Group：consumer接受数据的时候是按照consumer group来接受的，kafka确保每个partition只能同一个group中的同一个consumer消费，如果想要重复消费，那么需要其他的组来消费。对于一个topic，会广播给订阅此Topic的不同group。在group中，同时只有一个consumer可以消费消息

2：Broker：broker是kafka集群中的一个kafka节点, kafka没有单节点概念，也就是说一个节点的kafka也是集群模式

3：Topic：Kafka消息的类别，对数据进行切分和隔离

4：Partition：Kafka下数据存储的基本单元，一个Topic数据，会被分散存储到多个Partition中，多个Partition是并列无序的，每一个Partition内部的Message是有序的。
问题：为什么从Kafka消费的日志数据，永远是时序的呢？
答案：虽然多个Partition是并列无序的，但是消费者消费的时候，只要有一个partition有数据，就

5：Replication：副本，同一个Partition可能会有多个Replica，多个Replica之间数据是一样的

6：Replication Leader：一个Partition的多个Replica中，需要一个Replication Leader负责与Producer和Consumer交互

7：Replica Manager：负责管理当前broker中的partition和replica

    1：每一个Topic被切分为多个Partitions

    • Kafka中一个消息包含的字段

    1：Offset：偏移量，记录Partition消息的偏移位置

2：Length：消息长度

3：CRC32：校验字段，用于校验字段的完整性

4：Magic：判断是否是Kafka的消息

5：attribute：消息属性，是枚举值

6：Timestamp：消费的时间戳

7：Key Length：key的长度

8：Key：key的数据

9：Value Length：value的长度

10：Value：值的长度

    • 集群修改配置
    # id可以从1开始，当然也可以是从0开始
broker.id=1
# 监听的IP地址
listeners=PLAINTEXT://192.168.100.246:9092
# 这个是borker进行网络处理的线程数，为CPU和核心数
num.network.threads=32
# 这个是borker进行I/O处理的线程数
num.io.threads=20
# 发送缓冲区buffer大小，数据不是一下子就发送的，先回存储到缓冲区了到达一定的大小后在发送，能提高性能
socket.send.buffer.bytes=102400
# kafka接收缓冲区大小，当数据到达一定大小后在序列化到磁盘
socket.receive.buffer.bytes=102400
# 这个参数是向kafka请求消息或向kafka发送消息最大字节数，这个值不能超过java的堆栈大小
socket.request.max.bytes=104857600
# 数据目录不会放在/tmp目录下
log.dirs=/tmp/kafka-logs
# 默认一个topic的partitions为5
num.partitions=5
# 日志恢复的线程数
num.recovery.threads.per.data.dir=1
# 默认的partition副本集为2
default.replication.factor=2
# kafka内部默认的__consumer_offsets，partition默认是50，这是是设置其partition的副本数
offsets.topic.replication.factor=2
# 事务主题的复制因子
transaction.state.log.replication.factor=1
# 覆盖事务主题的min.insync.replicas配置
transaction.state.log.min.isr=1
# 默认消息的最大持久化时间，168小时，7天，可以改为1天
log.retention.hours=168
# 这个参数是：因为kafka的消息是以追加的形式落地到文件，当超过这个值的时候，kafka会新起一个文件
log.segment.bytes=1073741824
# 每隔300000毫秒去检查上面配置的log失效时间（log.retention.hours=168 ），到目录查看是否有过期的消息如果有，删除
log.retention.check.interval.ms=300000
# zookeeper地址
zookeeper.connect=192.168.100.246:2181,192.168.100.180:2181,192.168.100.200:2181
# zookeeper连接超时
zookeeper.connection.timeout.ms=18000
# 在实际使用时，假设你预估你的所有consumer组成员加入需要在10s内完成，那么你就可以设置该参数=10000
# 如果你发现消费者无法加入集群，那么可以修改这个值
group.initial.rebalance.delay.ms=10000
# 是否允许删除topic
delete.topic.enable=true
# 消费者定期向broker发送心跳，如果超过这个时间，消费组就会将其消费的partition分配给其他消费者
group.min.session.timeout.ms=6000
group.max.session.timeout.ms=3000000

    # 三：Kafka Manager的Topic 指标

    # （一）相关指标的含义

    • Partitions 分区
    	Filebeat将日志数据推送到Kafka，默认的分区数为5，分区数可以在Kafka Manager中动态创建
    1
    • Brokers
    	当前Topic数据分布在多少broker上
    1
    • Brokers Spread
    	broker使用率，如kafka集群5个broker，但是某个topic只分布在了2个broker中，那么broker spread=2/5 * 100% = 40%
    1
    • Brokers Skew
    	Partition的倾斜率，如kafka集群9个broker，某topic有18个partition，正常每个broker应该2个partition。若其中有3个broker上的partition数>2，则broker skew:  3 / 9 = 33%
    1
    • Brokers Leader Skew
    	Leader partition是否存在倾斜，如kafka集群9个broker，某topic14个partition，则正常每个broker有2个leader partition。若其中一个broker有0个leader partition，一个有4个leader partition，则broker leader skew: (4 - 2) / 14 = 14%
    1
