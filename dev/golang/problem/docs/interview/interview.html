    # (四)：Go 有异常类型吗

    # (五)：什么是Goroutine

    • Goroutine 是与其他函数或方法同时运行的函数或方法。 Goroutines 可以被认为是轻量级的线程。 与线程相比，创建 Goroutine 的开销很小。 Go应用程序同时运行数千个 Goroutine 是非常常见的做法

    # (六)：如何高效地拼接字符串

    # (七)：什么是 rune 类型

    • ASCII 码字符通过一个字节就可以表示，Unicode是ASCII的超集，在Golang中使用rune类型可以表示Unicode字符，rune是int32的别名，一般占用4个字节。Golang 字符串的底层表示是 byte 数组，而非 rune 数组。例如下面的例子中 语 和 言 使用 UTF-8 编码后各占 3 个 byte，因此 len("Go语言") 等于 8，当然我们也可以将字符串转换为 rune 序列，那么此时 len("Go语言") 等于 4
    fmt.Println(len("Go语言")) // 8

fmt.Println(len([]rune("Go语言"))) // 4

    # (十)：defer 的执行顺序 和 是否可以修改函数return的返回值

    • defer遵循先进后出的原语，最后声明的 defer 语句，最先得到执行。defer修改return的返回值需要分两种情况

    • 不带变量名的返回值：虽然defer 在 return 语句之后执行，但在函数退出之前，defer 可以修改函数返回值，但是Go 会创建一个临时变量保存返回值，defer中修改的是局部变量，因此返回值是0

    func test() int {
	i := 0
	defer func() {
	    i ++
		fmt.Println("defer1")
	}()
	return i
}

func main() {
	fmt.Println("return", test())
}

// 输出结果
defer1
return0

    # (十一)：Go 语言 tag 的用处

    • tag 可以理解为 struct 字段的注解，可以用来定义字段的一个或多个属性。框架/工具可以通过反射reflect获取到某个字段定义的属性，采取相应的处理方式。tag 丰富了代码的语义，增强了灵活性。下面代码中：json.Marshal将结构体进行序列化，Marshal函数内部使用了reflect获取到了json这个tag对应的字段值，因此可以将结构体的数据映射到tag的字段中
    package main

import "fmt"
import "encoding/json"

type Stu struct {
	Name string `json:"stu_name"`
	ID   string `json:"stu_id"`
	Age  int    `json:"-"`
}

func main() {
	buf, _ := json.Marshal(Stu{"Tom", "t001", 18})
	fmt.Printf("%s\n", buf)
}

    # (十二)：如何判断 2 个字符串切片（slice) 是相等的

    • slice类型是不可比较类型，但是Golang 语言中可以使用反射 reflect.DeepEqual(a, b)来判断两个变量的值是否相等，通常不推荐这么做，使用反射非常影响性能。一般在开发中，使用遍历的方式比较slice是否相等
    func StringSliceEqualBCE(a, b []string) bool {
    if len(a) != len(b) {
        return false
    }

    if (a == nil) != (b == nil) {
        return false
    }
	// b截取与a相等的长度
    b = b[:len(a)]
    // 遍历比较每个元素
    for i, v := range a {
        if v != b[i] {
            return false
        }
    }

    return true
}

    # (十三)：字符串打印时，%v 和 %+v 的区别

    • %v 和 %+v 都可以用来打印 struct 的值，区别在于 %v 仅打印各个字段的值，%+v 还会打印各个字段的名称，但如果结构体定义了 String() 方法，%v 和 %+v 都会调用 String() 覆盖默认值。
    type Stu struct {
	Name string
}

func main() {
	fmt.Printf("%v\n", Stu{"Tom"}) // {Tom}
	fmt.Printf("%+v\n", Stu{"Tom"}) // {Name:Tom}
}

    # (十六)：go slice 和 array 区别

    • 主要区别如下
    1：array是值类型，slice是引用类型，slice底层是引用了array存储数据，在作为函数参数传递的时候，array会重新拷贝一份

2：array的元素数据是一块连续的内存空间，slice类似一个动态的数组，在slice.go源码中，可以发现当slice容量不够的时候，按照原始容量的两倍规则自动扩容

3：在开发工作中，slice查询数据时间复杂度为O(1)，如果是删除元素，后面的元素都需要向前移动，时间复杂度为O(n)，如果对性能要求较高使用链表、双链表等线性数据结构

    # 二：实现原理

    # (一)：init() 函数是什么时候执行的

    • init() 函数是 Go 程序初始化的一部分。Go 程序初始化先于 main 函数，由 runtime 初始化每个导入的包，初始化顺序不是按照从上到下的导入顺序，而是按照解析的依赖关系，没有依赖的包最先初始化

    • 每个包首先初始化包作用域的常量和变量（常量优先于变量），然后执行包的 init() 函数。同一个包，甚至是同一个源文件可以有多个 init() 函数。init() 函数没有入参和返回值，不能被其他函数调用，同一个包内多个 init() 函数的执行顺序不作保证。一句话总结： import –> const –> var –> init() –> main()

    package main

import "fmt"

func init()  {
	fmt.Println("init1:", a)
}

func init()  {
	fmt.Println("init2:", a)
}

var a = 10
const b = 100

func main() {
	fmt.Println("main:", a)
}
// 执行结果
// init1: 10
// init2: 10
// main: 10

    # (二)：Go 语言的局部变量分配在栈上还是堆上

    • 由编译器决定。Go 语言编译器会自动决定把一个变量放在栈还是放在堆，编译器会做逃逸分析(escape analysis)，当发现变量的作用域没有超出函数范围，就可以在栈上，反之则必须分配在堆上

    • 例如在下面的foo() 函数中，如果 v 分配在栈上，foo 函数返回时，&v 就不存在了，但是这段函数是能够正常运行的。因为函数返回时，编译器发现 v 的引用脱离了 foo 的作用域，会将其分配在堆上。因此，main 函数中仍能够正常访问该值

    func foo() *int {
	v := 11
	return &v
}

func main() {
	m := foo()
	println(*m) // 11
}

    # (三)：2 个 interface 可以比较吗

    • Go 语言中，interface 可以使用 == 或 != 比较，interface 的内部实现包含了 2 个字段，类型 T 和 值 V。2 个 interface 相等有以下 2 种情况
    1：两个 interface 均等于 nil（此时 V 和 T 都处于 unset 状态）

2：类型 T 相同，且对应的值 V 相等。
    1
    2
    3
    • 下面例子中，stu1 和 stu2 对应的类型是 *Stu，值是 Stu 结构体的地址，两个地址不同，因此结果为 false。stu3 和 stu4 对应的类型是 Stu，值是 Stu 结构体，且各字段相等，因此结果为 true
    type Stu struct {
	Name string
}

type StuInt interface{}

func main() {
	var stu1, stu2 StuInt = &Stu{"Tom"}, &Stu{"Tom"}
	var stu3, stu4 StuInt = Stu{"Tom"}, Stu{"Tom"}
	
	fmt.Println(stu1 == stu2) // false
	fmt.Println(stu3 == stu4) // true
}

    # (四)：两个 nil 可能不相等吗

    • 这是可能的。Golang的interface是对非接口值(例如指针，struct等)的封装，interface内部包含 2 个字段，类型 T 和 值 V，当一个接口等于 nil，当且仅当 T 和 V 处于 unset 状态（T=nil，V is unset）。那么当nil指针赋值给interface变量，此interface就不是nil了。在下面的代码中， nil 的非接口值 p 赋值给接口 i，此时ii的内部字段为(T=*int, V=nil)，那么i 与 p 作比较时，将 p 转换为接口后再比较。p与nil比较只比较值，所以为true，nil与i比较时候，nil先转换为接口(T=nil, V=nil)，与i(T=*int, V=nil)就不相等了，因为T不相等
    func main() {
	var p *int = nil
	var i interface{} = p
	fmt.Println(i == p) // true
	fmt.Println(p == nil) // true
	fmt.Println(i == nil) // false
}

    # (五)：Golang的GC原理

    • 最常见的垃圾回收算法有标记清除(Mark-Sweep) 和引用计数(Reference Count)，Go 语言采用的是标记清除算法，并在此基础上使用了三色标记法和写屏障技术，提高了效率。(Python使用的主要是引用计数的GC回收方式)

    • 标记清除收集器是跟踪式垃圾收集器，其执行过程可以分成标记（Mark）和清除（Sweep）两个阶段：

    1：标记阶段：从根对象出发查找并标记堆中所有存活的对象

2：清除阶段：遍历堆中的全部对象，回收未被标记的垃圾对象，并将回收的内存加入空闲链表

    • 但是标记期间需要暂停程序(Stop the world)，待标记之后，程序才可以继续执行，为了解决这个问题，降低Stop the world的时间，Golang在标记算法的基础上，引入了三色标记算法
    三色标记算法将程序中的对象分成白色、黑色和灰色三类：

1：白色：不确定对象。
2：灰色：存活对象，子对象待处理。
3：黑色：存活对象。
    1
    2
    3
    4
    5

    • 标记开始时，所有对象加入白色集合（这一步需 Stop the world ）。首先所有将根对象标记为灰色，加入灰色集合，那么灰色集合中此时全部是根对象，然后垃圾搜集器取出一个灰色对象，将其标记为黑色，加入到黑色集合，并将其指向的对象标记为灰色，加入灰色集合，此时灰色集合中有根对象和子对象。重复这个过程，直到灰色集合为空为止，标记阶段结束。那么白色集合中剩下的对象就是需要清理的对象，而黑色对象均为根可达的对象，不能被清理。

    • 三色标记法因为多了一个白色的状态来存放不确定对象，所以后续的标记阶段可以并发地执行。当然并发执行的代价是可能会造成一些遗漏，因为那些早先被标记为黑色的对象可能目前已经是不可达的了。所以三色标记法是一个 false negative（假阴性）的算法。在 GC 过程中，对象指针发生了改变。比如下面的例子：

    A (黑) -> B (灰) -> C (白) -> D (白)
    1
    • 正常情况下，D 对象最终会被标记为黑色，不应被回收。但在标记和用户程序并发执行过程中，用户程序删除了 C 对 D 的引用，而 A 获得了 D 的引用。标记继续进行，D 就没有机会被标记为黑色了（A 已经处理过，这一轮不会再被处理）
    A (黑) -> B (灰) -> C (白) 
  ↓
 D (白)
    1
    2
    3

    • 为了解决这个问题，Go 使用了内存屏障技术，它是在用户程序读取对象、创建新对象以及更新对象指针时，执行的一段代码，类似于一个钩子。垃圾收集器使用了写屏障（Write Barrier）技术，当对象新增或更新时，会将其着色为灰色。这样即使与用户程序并发执行，对象的引用发生改变时，垃圾收集器也能正确处理了。

    • 一次完整的 GC 分为四个阶段

    1）标记准备(Mark Setup，需 STW)，打开写屏障(Write Barrier)，并发标记

2）使用三色标记法标记（Marking, 并发）

3）标记结束(Mark Termination，需 STW)，关闭写屏障。

4）清理(Sweeping, 并发)

    # (六)：函数返回局部变量的指针是否安全

    • 这在 Go 中是安全的。Go 编译器将会对每个局部变量进行逃逸分析，当发现变量的作用域超出函数范围，就会将局部变量放在堆上。栈内存用来存储局部变量和函数调用，可以理解为私有内存，堆内存中存放全局对象、结构体对象、成员变量，堆上的对象对所有线程可见，所有线程都可以访问堆上的对象，堆上的内存会被GC处理，因此用户返回局部变量是安全的

    # (七)：非接口的任意类型 T() 都能够调用 *T 的方法吗？反过来呢？

    • 这是一个指针接收者和值接收者的问题。类型T的值可以调用类型*T申明的方法，仅当此T的值是可寻址(addressable) 的情况下，值接收者调用指针接收者方法的时候，会先先取地址，将值接收者转换为指针。类型*T的值也可以调用类型T申明的方法，因为引用指针调用方法总是合法的，且对于每一个为类型 T 声明的方法，编译器都会为类型*T自动隐式声明一个同名和同签名的方法

    • 但是当类型T值和类型*T值赋值给接口变量的时候是有区别的，当使用类型*T实现的接口方法，必须使用类型*T的值赋值给接口变量，使用类型T的值赋值给接口变量就会报错

    # (八)：简要介绍Golang的GMP模型与Python协程调度的区别

    • GPM是Go语言运行时（runtime）层面的实现，是go语言自己实现的一套调度系统。对于Python而言，coroutine是自己控制调度的，主要是需要维护一个futurn可等待对象，当前任务没有完成的时候，通过yield关键字让出CPU，继续执行其他任务，当任务执行完毕后，外部的循环会再次执行future可等待对象，如果判断当前任务执行完毕，那么就会向下执行
    1：G就是goroutine，每次执行go func()都代表一个G，G包含了goroutine 的状态、堆栈、上下文、所绑定的P信息

2：P是Processor，管理着一组本地的goroutine队列，P会对自己管理的goroutine队列做一些调度，当自己本地的goroutine队列空了，会从其他的P管理的队列中获取一半的G，如果没有，会从全局的goroutine队列中获取。当P存放不下(超过256个)或者找不到空闲P情况下，会将goroutine放在全局goroutine队列中。P进行syscall的时候，不会将goroutine调度到其他的M。P决定了并行任务的数量，P的个数可以通过runtime.GOMAXPROCS来设置

3：M（machine）是Go运行时（runtime）对操作系统内核线程的虚拟，M与内核线程一般是一一映射的关系，一个groutine最终是要放到M上执行


单从线程调度讲，Go语言相比起其他语言的优势在于OS线程是由OS内核来调度的，goroutine则是由Go运行时（runtime）自己的调度器调度的。调度器使用m:n调度的技术（将n个goroutine到m个OS线程上执行），但是并发能力取决于P的个数

goroutine的调度是在用户态下完成的，不涉及内核态与用户态之间的频繁切换，包括内存的分配与释放，都是在用户态维护着一块大的内存池， 不直接调用系统的malloc函数，成本比调度OS线程低很多

充分利用了多核的硬件资源，近似的把若干goroutine均分在物理线程上， 再加上本身goroutine的超轻量，以上种种保证了go调度方面的性能

    # (一)：无缓冲的 channel 和 有缓冲的 channel 的区别

    # (二)：什么是协程泄露(Goroutine Leak)

    # (三)：Go 可以限制运行时操作系统线程的数量吗

    • 由Golang的GPM调度模型可知，可以使用环境变量 GOMAXPROCS 或 runtime.GOMAXPROCS(num int) 设置P的数量，由于P与M一一对应，那么OS线程数就是P的数量

    • 因此对于 CPU 密集型的任务，若该值过大，例如设置为 CPU 逻辑核数的 2 倍，会增加线程切换的开销，降低性能。对于 I/O 密集型应用，适当地调大该值，可以提高 I/O 吞吐率

    # (四)：go channel close后读的问题

    • Golang中的chan应用场景，一般在多个goroutine之间基于通信的方式完成数据共享。Golang中提倡通过通信共享内存，而不是通过共享内存实现通信，chan是goroutine之间通信的桥梁。

    • Golang的chan提供了FIFO的原语，保证了数据发送顺序，并且是并发安全的。chan的关闭不是必须的，chan可以被GC回收，当数据发送完毕，且通知接收者goroutine，chan才需要被关闭。

    • 在chan内部，其实是创建了一个hchan的结构体，结构体有几个属性，包括buf、sendx、recvx、lock等。从这里可以看出chan内部同样是使用了互斥锁来解决并发问题，但是用户在使用的时候，无法感知到加锁过程。chan内部是一个环形队列（ringbuffer）的实现，内部元素是以指针的形式存放在堆上。当chan的数据填充满了buf，那么发送者goroutine就会处于waiting状态，接收者goroutine从chan中获取到数据后，会再次唤醒发送者goroutine

    1：对一个关闭的通道再发送值就会导致panic。

2：关闭一个已经关闭的通道会导致panic。

3：对一个关闭的通道进行接收会一直获取值直到通道为空。

4：对一个关闭的并且没有值的通道执行接收操作会得到【对应类型】的零值。
