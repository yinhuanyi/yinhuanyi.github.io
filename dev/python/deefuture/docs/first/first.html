    # （三）在Django项目中，给apps设置为Sources Root

    	由于在apps包里面，会创建很多的app， 例如students。如果你想在项目的任何一个文件随时导入一个app里面的文件，那么需要将apps作为Sources Root， 那么pycharm将会知道，文件的根路径为apps。
	
	例如： from students import views ， pycharm是不会报错的。都是如果你使用命令行运行这个Python文件，那么就会报错，因为Python解释器认为：Python模块的搜索路径是从当前运行的Python文件所在目录开始寻找，然后在是项目的根目录, 一级一级找，最后到site-packages目录(第三方包目录)。

	那么如果你想使用命令行运行这个文件不报错，且将apps在命令行模式下也是Sources Root的话，需要在settings.py文件中加入一行：
	sys.path.insert(0, os.path.join(BASE_DIR, 'apps'))。  
	目的就是在django的sys.path模块搜索路径中的第一个位置加上apps的路径：/Users/yinhuanyi/Python_Virtual_Env_2.7.10/Djang_test/apps

    # （八）在settings中指定静态文件static的路径, 这样就能够直接在template中指定static文件的相对路径

    1：打开settings文件，你会发现已经指定了static为/static/， 
STATIC_URL = '/static/'

2：但是在template中是无法使用这个路径的，因为没有把这个路径加入到项目的路径中，因此需要把static路径加入到项目的路径中
STATICFILES_DIRS = (os.path.join(BASE_DIR, "static"), )  # 等价于 /Users/yinhuanyi/Python_Virtual_Env_2.7.10/MxOnline/static

3: 那么在template中引用这个静态文件就可以这样引用，指定相对路径就好了
<link rel="stylesheet" type="text/css" href="{% static 'css/muke/course/learn-less.css' %}"/###

    # （十一）Django中的post表单提交操作

    	1：在Django中的每一次表单提交操作都需要提供一个{% csrf_token %}, 因为每次get这个页面的时候，服务器端都会在浏览器端写入token到cookie中。当用户点击提交请求，post表单的时候，这个 {% csrf_token %} 就会做变量替换，替换为一个input标签，获取到浏览器的token，提交到服务器端，服务器端会先通过Django的CSRF token的浏览器，验证token是否是合法的，如果合法，那么请求会通过拦截器，到达服务器端的view处理逻辑，然后就可以取出用户提交的数据了

	2：Django中还有一种csrf认知机制，就是通过ajax，将token写入http的请求头。通过ajax完成一个http的自定义请求头

	3：Django没有referer的CSRF认证机制，referer的认证是最简单的一种，确保每次表单提交的源站都是后台允许的源站

    # （十二）Django中的object管理器

    	每次当model类调用object去处理数据的时候，都会返回一个Queryset对象，这个Queryset就像一个列表一样，需要遍历才能得到每一条记录对象
	
	object常用的方法有：
		all(): 返回表中的所有的行
		filter()：返回指定的行，可以对字段进行过滤
	
	记录对象可以实现的操作有：
		message.name: 获取message对象的name字段的值
		message.save: 将message对象写入表中
		message.delete: 将将message对象从表中删除

    # （十四）简单介绍Django中的过滤器

    使用的基本方式为：{{ 变量名称|过滤器名称 }}
1：例如：{{ value|add:"2" }} ： 如果 value 为 4,则会输出 6

2：例如：{{ first|add:second }}：first 是 [1, 2, 3] ，second 是 [4, 5, 6], 将会输出 [1, 2, 3, 4, 5, 6]， 同样适合于字符串的相加

3：例如：{{ value|cut:" " }}：如果value为“String with spaces”，输出将为"Stringwithspaces"。

    # （十五）为Django自建的user表进行扩展

    # 使用AbstractUser， 这样可以使用Django自带的用户字段，也可以添加我们自定义的字段，AbstractUser 中的所有的字段就是auth.user表中的字段，因此扩展的时候直接继承AbstractUser这个model就好了

from django.contrib.auth.models import AbstractUser
class UserProfile(AbstractUser):
    nick_name = models.CharField(max_length=50, verbose_name=u'昵称', default=u"")
    # 这里的null 和blank一般是一组合，说明用户可以不填写
    birday = models.DateField(verbose_name=u"生日", null=True, blank=True)
    # 对于这个性别，有两种类型，所以使用枚举
    gender = models.CharField(
        # 对应后台管理系统中的选择框
        choices=(("male", u"男"), ("female", u"女")),
        default="female",
        max_length=50,
    )

    address = models.CharField(max_length=100, default=u"")
    mobile = models.CharField(max_length=11, default=u"")
    # 图片字段，有上传的能力
    image = models.ImageField(upload_to="image/%Y/%m", default=u"image/default.png", max_length=100)

    # （十六）对Django的Django.contrib是啥？做出解释

    1、它是一个强大的功能包，是Django的标准库。
2、Django的标准库存放在 django.contrib 包中。每个子包都是一个独立的附加功能包。 这些子包一般是互相独立的，不过有些django.contrib子包需要依赖其他子包。
3、在 django.contrib 中对函数的类型并没有强制要求 。其中一些包中带有模型（因此需要你在数据库中安装对应的数据表），但其它一些由独立的中间件及模板标签组成。
4、django.contrib 开发包共有的特性是: 就算你将整个django.contrib开发包删除，你依然可以使用 Django 的基础功能而不会遇到任何问题。

5、当 Django 开发者向框架增加新功能的时，他们会严格根据这一原则来决定是否把新功能放入django.contrib中。

from django.shortcuts import render

from django.contrib.auth.backends import ModelBackend

from django.db.models import Q



# 重写认证方法
class CustomBackend(ModelBackend):
    def authenticate(self, username=None, password=None, **kwargs):
        try:
            # 通过用户传递进来的用户名或邮箱判断用户是否是注册用户, 这里引入的Django的或查询方式Q， 这样可以实现与和或运算的随意书写
            user = UserProfile.objects.get(Q(username=username) | Q(email=username))
            # 使用Django的密码认证机制，看用户的密码是否能够匹配
            if user.check_password(password): # 如果匹配，把这个查询的用户return回去
                return user
            return None  # 匹配不了，返回None
        except Exception as e:  # 如果查询异常，返回None
            return None

class LoginView(View):
    def get(self, request):
        return render(request, "login.html", {})

    def post(self, request):
        # 引入form对input的信息做验证
        login_form = LoginForm(request.POST)
        if login_form.is_valid(): # 如果填入的值合法，开始查询数据库

            user_name = request.POST.get('username', "")
            pass_word = request.POST.get('password', "")

            user = authenticate(username=user_name, password=pass_word)

            if user:  # 如果数据库匹配正确
                if user.is_active: # 如果用户的账号是激活的
                    # login()函数，功能是：当用户第一次登录成功，那么给django_session表中会添加一条session记录，且在浏览器端写入一个sessionID，将index.html渲染给用户，且渲染过程中会判断{% if request.user.is_authenticated %}, 如果是True，那么显示用户登录状态信息。当用户第二次访问首页的时候，会带着浏览器的sessionID，发起请求，Django的拦截器会查询数据库，看看是否有这个sessionID存在，如果有，直接显示用户的登录信息
                    login(request, user) # 这里也是的如果用户没有登入的情况下，访问个人中心，自动跳转到登入页面
                    return HttpResponseRedirect(reverse('index'))
                else:

    # （四十七）使用Django 的login函数实现session保持的原理

    • HTTP请求无状态

    • HTTP请求有状态（给浏览器写入sessionID, 无法跨越存储）

    • cookie 与 session的区别
    cookie机制：
	当用户第一次成功登录的时候，服务器端给浏览器写入一段cookie信息，cookie信息中保存了用户的用户名和密码等认证信息，当用户第二次登录的时候，浏览器就会携带用户的cookie信息进行登录。因此，这样就导致不安全，因为浏览器的cookie信息可能被盗用，使得用户的认证信息被泄露了

session机制：
	正是由于cookie验证机制的不安全性，因此session机制诞生了
	session机制其实就是在cookie机制的基础之上，使得cookie的值为一段hash乱码，不再是以前的用户的认证信息了，当用户第一次请求的时候，首先会被settings文件中的'django.contrib.sessions' 这个APP 和 'django.contrib.sessions.middleware.SessionMiddleware'这个中间件所拦截，目的就是检查浏览器携带的sessionID的值是否与在django_session表中存在，如果存在那么取出sessionID在数据库中被加密的值，解密后，获得用户的登录信息，再从users.profile表中查询，如果查询到了，说明用户sessionID是合法的，那么在前端页面可以这样判断{% if request.user.is_authenticated %}，让页面显示用户已经登录的状态。如果浏览器没有携带用户的sessionID,或者与数据库中的sessionID不匹配，那么显示用户未登录状态。
	当用户第一次登录，点击登录按钮，会发起一个post请求到服务器端，如果登录成功，那么返回登录首页，同时，也会被settings文件中的'django.contrib.sessions' 这个APP 和 'django.contrib.sessions.middleware.SessionMiddleware'这个中间件所拦截，将用户的登录信息加密之后，写入django_session表中，然后返回一个sessionID对应的hash乱码给浏览器端。
	当用户第二次访问的时候，就会携带这个sessionID对应的hash乱码请求服务器，那么经过浏览器之后，就知道用户是合法用户还是非合法用户了
