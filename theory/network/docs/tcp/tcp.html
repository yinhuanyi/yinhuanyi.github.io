    # （一）tcpdump 和 Wireshark 有什么区别？

    • tcpdump 和 Wireshark 就是最常用的网络抓包和分析工具，更是分析网络性能必不可少的利器。
    1: tcpdump 仅支持命令行格式使用，常用在 Linux 服务器中抓取和分析网络包。

2: Wireshark 除了可以抓包外，还提供了可视化分析网络包的图形页面。

3: 所以，这两者实际上是搭配使用的，先用 tcpdump 命令在 Linux 服务器上抓包，接着把抓包的文件拖出到 Windows 电脑后，用 Wireshark 可视化分析。当然，如果你是在 Windows 上抓包，只需要用 Wireshark 工具就可以。

    # （二）tcpdump命令行

    • 选择类
    选项 示例 说明
    -i tcpdump -i eth0 指定网络接口，默认是0接口，any表示所有网卡
    -nn tcpdump -nn 不解析IP地址和端口号的名称
    -c tcpdump -c 5 限制要抓取的网络包的个数
    -w tcpdump -w file.pcap 将抓包数据保存到文件中，文件名称以.pcap结尾
    • 过滤类
    选项 示例 说明
    host、src host、dest host tcpdump src host 172.18.157.35 -nn 指定IP过滤
    port、src port、dest port tcpdump src port 80 -nn 指定端口过滤
    ip、ip6、arp、tcp、udp、icmp tcpdump tcp -nn 指定协议过滤
    and、or、not tcpdump host 172.18.157.35 and port 80 逻辑表达式
    tcp[tcpflages] tcpdump -nn "tcp[tcpflags] & tcp-syn !=0" 指定syn状态的TCP包

    # （三）tcpdump命令行保存到文件中，在wireshark中分析

    • tcpdump -i eth0 icmp and src host 172.18.157.35 -nn -w ping.pcap

    • 使用wireshark将ping.pcap文件打开，在其下面的网络包详情中，可以更清楚的看到，这个网络包在协议栈各层的详细信息。比如

    1: 可以在数据链路层，看到 MAC 包头信息，如源 MAC 地址和目标 MAC 地址等字段

2: 可以在 IP 层，看到 IP 包头信息，如源 IP 地址和目标 IP 地址、TTL、IP 包长度、协议等 IP 协议各个字段的数值和含义

3: 可以在 ICMP 层，看到 ICMP 包头信息，比如 Type、Code 等 ICMP 协议各个字段的数值和含义

    # （二）Wireshark 时序图的显示方式

    • Wireshark 可以用时序图的方式显示数据包交互的过程, 从菜单栏中，点击 统计 (Statistics) -### 流量图 (Flow Graph)，然后，在弹出的界面中的「流量类型」选择 「TCP Flows」，你可以更清晰的看到，整个过程中 TCP 流的执行过程

    • 我们都知道 HTTP 是基于 TCP 协议进行传输的，那么：
    1: 最开始的 3 个包就是 TCP 三次握手建立连接的包
2: 中间是 HTTP 请求和响应的包
3: 而最后的 4 个包则是 TCP 断开连接的挥手包

    2

    • 这里的【流量图】中显示的seq序列号是从0开始的，这是Wireshark做了优化。可以选择任意一个流量包，点击右键，选择【协议首选项】，取消【Relative Seq】，那么流量图就会显示真实的seq序列号的值

    从上图可以看到，SYN和ACK表示的是控制位，而Seq表示的是初始化的序列号，Ack表示应答号.(这里要注意ACK和Ack的区别，ACK是控制位，Ack是应答号)
    1
    • 可见，客户端和服务端的序列号实际上是不同的，序列号是一个随机值。这其实跟我们书上看到的 TCP 三次握手和四次挥手很类似，作为对比，你通常看到的 TCP 三次握手和四次挥手的流程，基本是这样的：

    • 三次握手
    1: 客户端会随机初始化序号（client_isn），将此序号置于 TCP 首部的「序号」字段中，同时把 SYN 标志位置为 1 ，表示 SYN 报文。接着把第一个 SYN 报文发送给服务端，表示向服务端发起连接，该报文不包含应用层数据，之后客户端处于 SYN-SENT 状态。

2: 服务端收到客户端的 SYN 报文后，首先服务端也随机初始化自己的序号（server_isn），将此序号填入 TCP 首部的「序号」字段中，其次把 TCP 首部的「确认应答号」字段填入 client_isn + 1, 接着把 SYN 和 ACK 标志位置为 1。最后把该报文发给客户端，该报文也不包含应用层数据，之后服务端处于 SYN-RCVD 状态。

3: 客户端收到服务端报文后，还要向服务端回应最后一个应答报文，首先该应答报文 TCP 首部 ACK 标志位置为 1 ，【序列号】是第二次握手的应答号，也就是第一次握手的序列号+1，其次「应答号」是第二次握手的序列号，最后把报文发送给服务端，这次报文可以携带客户到服务器的数据，之后客户端处于 ESTABLISHED 状态。

第一次和第二次握手不能携带数据，第三次握手可以携带数据

    • 为什么是三次握手
    1：如果客户端发送第一次握手的数据包，很久之后才到达服务器端，此时可能客户端已经关闭了，当服务器端收到第一次握手的SYN=1报文的时候，会回复一个SYN=1、ACK=1报文，此时客户端收到报文后，会根据自己的上下文环境，给服务器端发送一个RST=1的报文，close服务器端的连接状态。如果是两次连接，那么服务器端的连接状态不会关闭，以至于消耗服务器端的资源

2：需要确认序列号，当客户端发送seq=101报文到服务器端，服务器会发送一个seq=10101,Ack=102到客户端，表明服务器收到报文，客户端又会回复一个seq=102，Ack=10102的报文到服务器端，表明客户端收到报文
    1
    2
    3
    • 四次握手
    1: 客户端打算关闭连接，此时会发送一个 TCP 首部 FIN=1、ACK=1的报文，seq=101010,Ack=1111001, 之后客户端进入 FIN_WAIT_1 状态。

2: 服务器端收到报文后，向客户端发送一个ACK=1，seq=1111001, Ack=101011的应答报文，服务端进入 CLOSED_WAIT 状态。

3: 客户端收到报文后进入FIN_WAIT_2 状态，当服务器端处理完最后数据后，向客户端发送一个FIN=1、ACK=1，seq=1111001、Ack=101011的报文，服务端进入 LAST_ACK 状态

4：客户端收到报文后，回一个 ACK=1，seq=101011、Ack=1111002报文，之后进入 TIME_WAIT 状态。服务器端收到报文就进入CLOSE 状态，客户端在经过 2MSL(最大分段生存期) 一段时间后，自动进入 CLOSE 状态
    1
    2
    3
    4
    5
    6
    7
    • 为什么要四次握手
    1：关闭连接时，客户端向服务端发送 FIN 时，仅仅表示客户端不再发送数据了但是还能接收数据。

2：服务器收到客户端的 FIN 报文时，先回一个 ACK 应答报文，而服务端可能还有数据需要处理和发送，等服务端不再发送数据时，才发送 FIN 报文给客户端来表示同意现在关闭连接。
    1
    2
    3
    • 为何一定要等2MSL？
    如果不等，释放的端口可能会重连刚断开的服务器端口，这样依然存活在网络里的老的TCP报文可能与新TCP连接报文冲突，造成数据冲突，为避免此种情况，需要耐心等待网络老的TCP连接的活跃报文全部死翘翘，2MSL时间可以满足这个需求（尽管非常保守）！

	6

    • 解释说明
    1：客户端发起 SYN 后，由于防火墙屏蔽了服务端的所有数据包，所以 curl 是无法收到服务端的 SYN、ACK 包，当发生超时后，就会重传 SYN 包

2：服务端收到客户的 SYN 包后，就会回 SYN、ACK 包，但是客户端一直没有回 ACK，服务端在超时后，重传了 SYN、ACK 包，接着一会，客户端超时重传的 SYN 包又抵达了服务端，服务端收到后，超时定时器就重新计时，然后回了 SYN、ACK 包，所以相当于服务端的超时定时器只触发了一次，又被重置了。

3：最后，客户端 SYN 超时重传次数达到了 6 次（tcp_syn_retries 默认值 6 次），就不再继续发送 SYN 包了。

4：所以，我们可以发现，当第二次握手的 SYN、ACK 丢包时，客户端会超时重发 SYN 包，服务端也会超时重传 SYN、ACK 包。

    # （二）高级包头过滤

    • 首先了解如何从包头过滤信息，proto表示协议的意思，ip[0]，表示取IP报文的第一个字节
    proto[x:y] : 过滤从x字节开始的y字节数。比如ip[2:2]过滤出3、4字节（第一字节从0开始排）
proto[x:y] & z = 0 : proto[x:y]和z的与操作为0
proto[x:y] & z !=0 : proto[x:y]和z的与操作不为0
proto[x:y] & z = z : proto[x:y]和z的与操作为z
proto[x:y] = z : proto[x:y]等于z

操作符 : >, <, >=, <=, =, !=

    • 在判断IP包头的时候，需要先判断是否包头设置了选项
    一般而言，IP头部是20字节大小(由上图可知)，但是如果IP头部设置了选项，那么IP报文的数据就不能从21字节处读取，IP头部有一个字段可以判断IP头部的长度：【首部长度】。因此在数据抓包的时候，可以抓取第一个字节，看看是不是01000101。由IP报文头可以知道，【版本】+ 【首部长度】= 8bit = 1Byte
01000101可以拆解为：0100 和 0101， 那么如果需要判断是否IP报文添加了选项，可以这样抓包：抓取没有选项的IP报文
tcpdump -i any 'ip[0] & 15 = 5' and src 192.168.40.224 and dst 192.168.201.249 -nn
